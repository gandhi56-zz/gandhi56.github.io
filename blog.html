Mini-battleship

Mini-battleship is a problem on Open-kattis which was hosted in the ICPC North American Championship 2020. If you have ever played the game of battleship before, the solution to this program might be exactly what you would like to have with you when you are playing this game in reality. Nevertheless, I can briefly explain the problem here:

Problem

Battleship is a two-player board game, each player has their own grid which is hidden from their opponent. Each player secretly places some ships on their grid, each ship covers horizontal or vertical straight line of one of more contiguous squares without overlappings. Each player takes turns taking shots at their opponent's ships by calling out their opponent's grid and the opponent will say whether the shot was a hit or a miss.

We are given an n x n grid, where 1 <= n <= 5, of what we have seen in our opponent's grid so far. Each cell in the grid can either be:

  • X representing a miss
  • O representing a hit
  • . representing a square where no shot has been taken yet
We are also given the number of ships and their sizes our opponent has placed on their grid. Our goal is to write a program that counts the number of possible configurations with all ships placed in the grid and our shots are satisfied in each configuration.

My Experience solving this problem

Observing how small the value of n can be and comparing it with the time limit of 6 seconds, I was confident that a brute force approach would give me the correct solution. So I implemented my solution, debugged it, tested it, debugged it again and tested it again. Everything seemed normal to me since this is exactly what happens when I am solving such brute force problems. No surprises so far! Until I reached a point where I satisfied two of the three sample inputs given. All three test cases are not trivial at all! I read my code multiple times, no clue appeared to me as to why I was not satisfying all three test cases or even why only the first and third test cases were satisfied.

I informed my competitive programming coach about this and during our discussion, I realized that I swapped X's and O's in my code! That is, my code imagined that X's are supposed to be hits and O's are supposed to be misses. I swapped them immediately after realizing and tested against the sample inputs. All sample inputs satisfied! Without any further testing only because I was confident enough about my solution, I submitted my code to the judging system and my solution was accepted! It was an interesting experience trying to get my code accepted for this problem mainly because the sample test cases were designed so deceptively.
